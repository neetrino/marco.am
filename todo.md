## Шаг 4 — Канонические одноязычные URL (Вариант 2)
Цель: убрать локализацию из ссылок (слаги категорий/товаров — один канонический язык), при сохранении многоязычного контента.
- [ ] Категории: использовать один канонический слаг в URL независимо от языка интерфейса (резолв `slug → categoryId` уже locale‑agnostic).
- [ ] Товары: один канонический слаг в URL (в `ProductPdpRow.slugs` уже хранятся все слаги — lookup не сломается).
- [ ] Эффект: стабильные URL → один ключ кеша/префетча вместо трёх, лучше SEO (canonical), надёжнее prefetch.
- [ ] Важно: сам по себе НЕ делает страницу статикой и НЕ убирает `cookies()`. Это дополнение к кешу/статике, не замена.
- [ ] Риск: средний (меняются публичные URL → нужны redirect'ы со старых локализованных слагов, проверить SEO/шаринг).

## Шаг 5 — Статическая оболочка на языке по умолчанию + cookie‑язык постфактум (Вариант 3)
Цель: снять `cookies()` с серверного критического пути → сайт отдаётся статикой/из CDN, мгновенный первый кадр.
- [ ] Рендерить статическую оболочку на языке по умолчанию (без `cookies()` в `layout.tsx` и контексте `/products`).
- [ ] Применять cookie‑язык на клиенте ДО первого кадра инлайн‑скриптом `beforeInteractive` (как уже сделано для темы `theme-init`) → почти без мигания для UI‑текста.
- [ ] Локализованный серверный контент (категории/мета) в исходном HTML будет на языке по умолчанию → оценить влияние на SEO.
- [ ] Это главный рычаг полной статики. Делать только если кеш (Шаг 3) по замерам недостаточно.
- [ ] Риск: высокий (i18n + SEO + возможное мигание). Обсудить до начала.
- [ ] Альтернатива «золотой стандарт»: локаль в пути (`/en`, `/hy`, `/ru`) + `generateStaticParams` — статика по локали, идеальный SEO, без мигания, но смена структуры URL.
